## 二、三条命令

### 命令 1：QC 清洗检查

```bash
python clean.py
```

**输入**：`E:/datasets/egos/墨现/` 下全部原始文件
**输出**：`clean_report.txt`（终端 + 文件）
**10 步管线**：

| 步骤 | 检查项 | 模块 |
|------|--------|------|
| 1 | 读取 meta.json | `reader.py` |
| 2 | 统计 index.jsonl 帧数 | `reader.py` |
| 3 | 帧数一致性比较 | `timestamp_checker.py` |
| 4 | 时间戳间隔检测（>40ms） | `timestamp_checker.py` |
| 5 | 视频基本信息 | `video_checker.py` |
| 6 | 坏帧检测（解码失败） | `video_checker.py` |
| 7 | 黑屏检测（灰度<5） | `video_checker.py` |
| 8 | 读取 IMU | `reader.py` |
| 9 | IMU 中断检测（>60ms） | `imu_checker.py` |
| 10 | 生成清洗报告 | `report.py` |

当前样本结果：**PASS**（0 个异常，983 帧，32.77s）

### 命令 2：异常检测 → 分段方案

```bash
python -m zpds_prepare.main "E:/datasets/egos/墨现" --output output/
```

**输入**：原始数据（MKV + index.jsonl + IMU CSV）
**输出**：`output/quality_issues.json` + `output/segment_candidates.json`
**6 步管线**：

| 步骤 | 内容 | 模块 |
|------|------|------|
| ① 读取原始数据 | meta / index / IMU / MKV | `readers/guida_reader.py` |
| ② 运行检测器 | 黑屏 + 视频缺口 + IMU 缺口 | `detectors/` |
| ③ 汇总分析 | 按类型 / 按处置统计 | `decisions/segment_planner.py` |
| ④ 写出质量报告 | quality_issues.json | `writers/quality_writer.py` |
| ⑤ 生成候选 Segment | trim/split 决策 → 切分 | `decisions/segment_planner.py` |
| ⑥ 写出分段方案 | segment_candidates.json | `writers/candidate_writer.py` |

当前样本结果：**0 个异常**，**1 个候选**（覆盖完整 32.77s）

### 命令 3：批量生成 Prepared Segment

```bash
python batch_prepare.py
# 等效: python batch_prepare.py --candidates output/segment_candidates.json --output prepared_segments/
```

**输入**：`segment_candidates.json` + 原始数据
**输出**：`prepared_segments/seg_XXXXXX/` + `batch_summary.json`
**流程**：

```
① 读取 segment_candidates.json
② 预加载共享资源（index.jsonl + calibration）
③ 对每个候选区间：
   ├── 裁剪转码 RGB 视频 (MKV → MP4/H.264 CFR 30fps)
   ├── 生成 sample_map (最近邻映射)
   ├── 裁剪规范化 IMU (→相对时间 + 标准列名)
   ├── 提取 calibration (共享，只写一次)
   ├── 生成 segment.json
   └── 写出后验证 (6 项校验)
④ 汇总 → batch_summary.json
```

每个 Segment 耗时约 40-60 秒（主要花在 MKV 帧级读取转码）。

## 四、统一异常模型：QualityIssue

所有检测器的输出统一为此数据类，不再 `print()`：

```python
@dataclass
class QualityIssue:
    issue_type: str              # continuous_black_frames | timestamp_gap | imu_gap
    stream_id: str               # ego_rgb | ego_imu
    start_ns: int                # 异常起始 (设备时钟)
    end_ns: int                  # 异常结束 (设备时钟)
    severity: str                # warning | error | critical
    decision: str                # trim | split | keep_with_flag
    details: dict                # {"frame_count": 24, "gap_ns": 500000000, ...}
```

### 决策规则速查

| 检测器 | 条件 | decision |
|--------|------|----------|
| 黑屏 | 距首尾 ≤1s 的连续黑屏 ≥0.5s | `trim` |
| 黑屏 | 中间的长黑屏 ≥0.5s | `split` |
| 黑屏 | 中间短黑屏或不足阈值 | `keep_with_flag` |
| 视频缺口 | 间隔 >2× 正常 且 ≥0.5s | `split` |
| 视频缺口 | 间隔 >2× 正常 但 <0.5s | `keep_with_flag` |
| IMU 缺口 | 间隔 >3× 正常 且 ≥1.0s | `split` |
| IMU 缺口 | 间隔 >3× 正常 但 <1.0s | `keep_with_flag` |

## 五、候选 Segment 生成算法

`segment_planner.py` → `plan_segments()`

```
输入: issues (list[QualityIssue]), session_start_ns, session_end_ns

1. 分离 issue:
   - decision = "trim" 且靠近起始 → head_trims
   - decision = "trim" 且靠近末尾 → tail_trims
   - decision = "split" → split_boundaries

2. 计算有效范围:
   valid_start = max(session_start, 最晚 head_trim.end)
   valid_end   = min(session_end,   最早 tail_trim.start)

3. 切分:
   按 split_boundaries 中的缺口切割 valid range
   [valid_start, gap1.start], [gap1.end, gap2.start], ...

4. 过滤:
   - 丢弃 duration < min_duration_s (默认 1s)
   - 标记 duration > max_duration_s (默认 120s)

5. 关联:
   每个候选关联落在其区间内的 keep_with_flag issues
```

# ---- 视频质量检测 ----
video:
  black_threshold: 5                    # (旧) 黑屏灰度阈值
  min_black_duration_s: 0.5             # (旧) 连续黑屏最短时长

    min_duration_s: 0.5                 #   最短持续时长
    edge_tolerance_s: 1.0               #   首尾容差

# ---- 时间戳异常检测 ----
timestamp:
  video_gap_factor: 2.0                 # (旧) 视频缺口倍数
  imu_gap_factor: 3.0                   # (旧) IMU 缺口倍数
  video:                                # (新) 视频缺口分级
    gap_factor: 2.0                     #   检测阈值倍数
    split_gap_s: 0.5                    #   切分阈值 (秒)
  imu:                                  # (新) IMU 缺口分级
    gap_factor: 3.0                     #   检测阈值倍数
    split_gap_s: 1.0                    #   切分阈值 (秒)

# ---- 输出 ----
output:
  target_fps: 30.0                      # 输出 CFR 帧率

# ---- Segment 约束 ----
segment:
  min_duration_s: 1.0                   # 最短有效 Segment
  max_duration_s: 120.0                 # 最长有效 Segment

## 九、技术决策记录

### 为什么用 OpenCV 而不是 FFmpeg？
Windows 环境没有 FFmpeg。OpenCV 的 `VideoCapture`/`VideoWriter` 是唯一可移植方案。mp4v 编码器兼容性最好，avc1 (H.264) 在安装 openh264 DLL 后可用。

### 为什么 IMU 不做重采样？
保留原始 50Hz 采样节奏（标记为 `irregular`）。重采样会引入插值误差，下游算法应根据实际时间戳自行处理。

### 为什么不做去重合并？
IMU 的加速度计和陀螺仪数据分两行写入（相同 timestamp_ns，不同 gx/gy/gz 值），这是原始采集程序的写入方式。保留原始结构，不做合并。

### 为什么先检测后生成？
`zpds_prepare/` 产出 `segment_candidates.json` 之后、`batch_prepare.py` 运行之前，可以人工审核候选方案（确认切分位置是否合理），再批量生成。这是 Pipeline 中的一个可选的 Human-in-the-loop 节点。

### 为什么 sample_map 用 Parquet？
- 压缩率高（~10:1 相比 CSV）
- 列式存储，下游只需读特定列
- pandas/polars 原生支持，无需额外解析

### 为什么 segment.json 的 quality.issues 从 candidates 传入？
生成的候选 Segment 区间内可能包含 `keep_with_flag` 的 issue（如小缺口），这些 issue 不触发切分但需要标记在最终 Segment 中，供下游训练时参考（如跳过该帧或加权降权）。
