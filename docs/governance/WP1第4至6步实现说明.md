# WP1 第 4～6 步实现说明

本阶段在前三步的 Config 与 Stage 契约之上，补齐本地 Storage Adapter、Run Ledger，
以及支持幂等、重试和断点恢复的顺序 Runner。

## 4. 本地 Storage Adapter

`LocalStorage` 使用两个明确的逻辑命名空间：

- `raw://...`：指向不可变原始数据，只提供读取、存在性检查和流式 SHA256；
- `artifact://...`：指向 Ledger、报告和后续 Prepared 等派生产物。

```python
from zpds.storage import LocalStorage

storage = LocalStorage(
    raw_root="D:/datasets",
    artifact_root="D:/zpds-output",
)

with storage.open_read("raw://egos/session/index.jsonl") as file:
    header = file.readline()

digest = storage.sha256("raw://egos/session/index.jsonl")
```

安全边界：

- 拒绝绝对路径、反斜杠、盘符、空路径段和 `..`；
- 路径解析后必须仍位于对应根目录，已有 symlink 也不能借此越界；
- Raw 没有写入 API；
- Artifact 文件使用同目录临时文件，校验后 `os.replace`；
- Artifact 目录使用同目录临时目录，校验后一次发布；
- 写入或校验异常会清理临时产物，不留下“看起来已完成”的目标；
- 完整目录已存在时拒绝覆盖，避免破坏已发布 revision。

## 5. Run Ledger

`FileRunLedger` 将每个 run 保存为：

```text
<artifact_root>/runs/<run_id>/ledger.json
```

Ledger 使用 `run_ledger.schema.json` 校验，并记录：

- run、session、初始输入、配置哈希和代码版本；
- Stage 的编号、名称和版本；
- `pending`、`running`、`succeeded`、`failed`、`skipped` 状态；
- 确定性 execution key；
- 累计 attempt 次数；
- 输入、输出、开始/结束时间和错误；
- 完整 StageResult，包括 metrics、decisions 和 evidence。

相同 `run_id` 如果换了 session、输入、配置、代码版本或 Stage 集合，会明确抛出
`LedgerConflictError`，不会覆盖旧审计记录。

## 6. 幂等、重试和断点恢复 Runner

Runner 的 execution key 由以下内容规范化后计算 SHA256：

- Stage ID、名称和版本；
- session ID；
- 当前 Stage 的输入引用；
- config hash；
- code version。

`run_id` 不进入 execution key，但当前实现只在同一个 Run Ledger 中复用结果。

```python
from zpds.config import PipelineConfigLoader
from zpds.pipeline import FileRunLedger, PipelineRunner, StageContext
from zpds.storage import LocalStorage

config = PipelineConfigLoader().load("configs/pipeline/pilot.yaml")
storage = LocalStorage("D:/datasets", "D:/zpds-output")
ledger = FileRunLedger(storage)
runner = PipelineRunner((stage0, stage1, stage2), ledger)

result = runner.run(
    StageContext(
        run_id="guida-demo-001",
        session_id="session-001",
        input_refs=("raw://egos/session-001/index.jsonl",),
        config=config,
        code_version="git-commit-or-build-id",
    )
)
```

运行规则：

1. 只选择 `pipeline.stages` 中启用且调用方已经提供实现的 Stage；
2. 已有相同 execution key 的成功结果直接复用；
3. 上一 Stage 的输出成为下一 Stage 的输入；
4. 普通异常转成可审计的 failed StageResult；
5. 按 `runner.max_retries` 自动重试，并记录每一次 attempt；
6. `KeyboardInterrupt`、进程终止等中断会让 Stage 保持 `running`；
7. 同一 `run_id` 再次执行时，成功 Stage 被跳过，未完成 Stage 产生新 attempt。

默认配置：

```yaml
runner:
  max_retries: 1
  retry_backoff_seconds: 0
```

- Raw 读取、哈希与路径越界；
- Artifact 文件替换、目录发布和失败回滚；
- Ledger Schema、身份冲突与 StageResult 回读；
- 自动重试及 attempt 计数；
- Stage 输出向下游传递；
- 中断后续跑；
- 完成后重复执行全部复用。

- 跨进程锁和分布式并发调度；

- Pipeline CLI、结构化日志和指标导出；
